# Conventions Specification

This document defines the cross-repo workflow conventions that apply to all repositories using sdlc-workflow skills, and references the per-repo CONVENTIONS.md template for project-specific coding standards.

---

## Part 1 — Workflow Conventions

These conventions apply to all repositories and are enforced by the sdlc-workflow skills. See [docs/constraints.md](constraints.md) for the full set of deterministic rules.

### Commit Message Format

All commits follow the [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) specification:

```
<type>[optional scope]: <description>

[optional body]

Implements <JIRA-ID>
Assisted-by: Claude Code
```

**Rules:**
- **Type** is one of: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`
- **Scope** is optional but recommended when relevant (e.g., `feat(api): add AIBOM endpoint`)
- **Footer** MUST include the Jira issue ID (e.g., `Implements TC-123`)
- **Trailer** MUST include `Assisted-by: Claude Code` (added via `--trailer`)

### PR Workflow

1. **Branch naming:** The feature branch MUST be named after the Jira issue ID (e.g., `TC-123`)
2. **PR description:** MUST follow this format:

```
## Summary
- <bullet point describing a change>
- <bullet point describing a change>

## Jira
Implements [<JIRA-ID>](https://<jira-host>/browse/<JIRA-ID>)
```

   Each further commit on the same feature branch MUST update the Summary bullet points to reflect the current state of changes in the PR.

3. **Jira linking:** After opening a PR, its link MUST be posted as a comment on the Jira task
4. **Custom field:** If the project's Jira Configuration includes a Git Pull Request custom field, the PR URL is also set on that field (in ADF format)
5. **Transition:** The Jira task is transitioned to In Review after the PR is opened

### Task Template Structure

Tasks generated by `plan-feature` follow a structured description template. The following sections are **required** in every task:

- **Repository** — single repository per task
- **Description** — what the task achieves and why
- **Acceptance Criteria** — pass/fail checklist
- **Test Requirements** — tests to write or update

The following sections are **optional** and omitted when not applicable:

- Files to Modify, Files to Create, API Changes, Implementation Notes, Verification Commands, Dependencies

**Template rules:**
- File paths must be real paths discovered during repository analysis
- Implementation Notes must reference existing patterns found in the code
- Empty sections are omitted, not left blank

## Part 2 — Per-Repo CONVENTIONS.md Template

Each repository using the sdlc-workflow should have a `CONVENTIONS.md` file documenting project-specific coding standards. This helps the AI assistant follow your project's patterns when generating or modifying code.

The reference template is maintained as a skill asset at:

```
plugins/sdlc-workflow/skills/setup/conventions.template.md
```

The `/sdlc-workflow:setup` skill can use this template during onboarding to scaffold a `CONVENTIONS.md` file in the target repository.

### Suggested Sections

The template includes the following sections (see the [template file](../plugins/sdlc-workflow/skills/setup/conventions.template.md) for full content with examples):

- **Language and Framework** — primary languages, frameworks, and toolchains
- **Code Style** — formatting rules, linters, and style guides
- **Naming Conventions** — naming patterns for types, functions, files, and APIs
- **File Organization** — where new files should be placed
- **Error Handling** — error handling patterns and conventions
- **Testing Conventions** — testing patterns, frameworks, and coverage requirements
- **Commit Messages** — project-specific commit message conventions (extending the workflow conventions above)
- **Dependencies** — policies for adding or managing dependencies
